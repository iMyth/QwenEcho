Three independent C++ modules share a common shape: each owns an opaque handle struct (`ThermalMonitor`, `MemoryMonitor`, `LatencyTracker`) exposed through an `extern "C"` create/start/stop/get/destroy API. Two of them (thermal, memory) spawn a dedicated low-priority `std::thread` that loops on a `condition_variable::wait_for` with a fixed poll interval (5 s for thermal, 2 s for memory), reading values from the HAL (`hal_thermal_get_temperature`, `hal_memory_get_rss`) and transitioning a state machine with hysteresis before invoking a user callback and posting a `MSG_*` event through `native_port_post_*`. The latency tracker is not polled — it is a per-segment circular buffer (`kMaxTrackedSegments = 32`) keyed by `segment_id`, updated by stage lifecycle hooks (`on_asr_start/end`, `on_llm_first_token`, `on_tts_first_audio`) and evaluates stage/E2E budgets against a throttle mode flag set by the thermal monitor. All three modules depend only on their own header, `native_port.h`, and the relevant HAL; there is no cross-dependency between the monitors themselves.
