Three cooperating C-style components built on opaque `struct`s allocated with `calloc` and constructed via placement-new:
- `engine_manager.cpp` exposes `EngineManager`, a mutex-guarded state machine (`UNINITIALIZED → INITIALIZING → READY → RUNNING → STOPPING`) that owns a `ModelLoader` (GGUF models) and a `PipelineController`. It initializes the llama.cpp backend via `gguf_inference_backend_init()` before loading ASR/LLM/TTS models, then delegates session control to the controller.
- `pipeline_controller.cpp` exposes `PipelineController`, which wires the full cascade: `AudioCollector → AudioRingBuffer → SentenceSegmenter → AsrStage → LlmStage → TtsStage`, connected by two `BoundedSPSCQueue`s (`AsrToLlmElement`, `LlmToTtsElement`). A dedicated feeder thread reads 100ms chunks from the ring buffer into the segmenter. Monitors (`ThermalMonitor`, `MemoryMonitor`, `LatencyTracker`) observe system health; memory pressure at level 2 triggers a graceful stop. Destruction follows reverse-pipeline order inside `destroy_pipeline_resources()`, bounded by a 2-second deadline with polling for queue drain.
- `engine.cpp` is currently a placeholder stub awaiting stage implementations.
Dependency direction: `EngineManager` → `PipelineController` → stages + monitors; both depend on `model_loader.h` and `gguf_inference.h` (llama.cpp wrapper). Cross-platform logging uses `os_log` on macOS and `fprintf(stderr)` elsewhere via an `ECHO_LOG` macro.
