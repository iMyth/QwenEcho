Two cooperating sub-layers share one goal: hide platform differences behind stable `extern "C"` headers.

- HAL interfaces (`native/hal/*.h`) declare opaque-handle + create/start/destroy lifecycles for five subsystems — accelerator, audio, memory, thermal, thread. Each header is paired with two platform implementations under `native/hal/android/` (`.c`) and `native/hal/ios/` (`.m`/`.c`), selected at link time by the host OS. The accelerator layer additionally accepts an in-memory GGUF blob and dispatches to NNAPI/Vulkan on Android or CoreML/MPS on iOS with CPU fallback.
- `native/src/native_port.cpp` is the only C++ entry point. It owns a single global `std::atomic<int64_t>` Dart port id plus an overridable `PostFn` function pointer (defaulting to `Dart_PostCObject_DL` from the Dart API DL). Typed helpers (`native_port_post_asr_partial`, `_translation_stream`, `_tts_started`, …) serialize payloads as fixed-layout `Dart_CObject[]` arrays whose first element is a `MSG_*` enum constant, then call the stored post function. Logging uses `os_log` on Apple targets and `fprintf(stderr)` elsewhere.
- Dependency direction is strictly downward: pipeline stages call into `hal_*` and `native_port_*`; the HAL never depends on higher layers. The module exposes no C++ symbols — everything is `extern "C"` so Dart FFI can bind directly.
