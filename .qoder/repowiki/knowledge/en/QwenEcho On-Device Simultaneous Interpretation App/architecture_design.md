The repository composes three layers around a single contract: the Flutter shell (`dart_app`) exposes typed `EchoMessage` streams consumed by both the view and status overlays; the iOS runner (`ios_platform`) boots Flutter and links the local `qwen_echo_engine` shared library; the C++ core (`native_engine`) implements the audio collector → ring buffer → VAD/segmenter → ASR → LLM → TTS → speaker pipeline behind four C-linkage entry points (`InitQwenEchoEngine`, `StartEchoPipeline`, `StopEchoPipeline`, `RegisterEchoMessagePort`). The Dart FFI bindings in `lib/src/native_bridge.dart` call into those symbols and push results back through a Native Port, while `dart_tests` validates the message protocol and UI components. Platform HALs under `native/hal/{android,ios}` isolate OS-specific audio, thermal, memory, and accelerator calls so the same engine binary targets both platforms. All model assets (FunASR-Nano, Qwen3-4B-Instruct, Qwen3-TTS-Streaming) are GGUF/INT4 files bundled as Flutter assets or loaded from the app sandbox at runtime — no network is used anywhere.
