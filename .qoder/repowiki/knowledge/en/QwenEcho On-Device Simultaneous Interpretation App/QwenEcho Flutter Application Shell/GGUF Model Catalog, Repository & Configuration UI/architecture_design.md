Two-layer leaf module with a clear separation of concerns:
- `model_catalog.dart` is a pure data layer exposing the immutable `ModelKind` enum (ASR/LLM/TTS in pipeline order) and the `kRequiredModels` list of `ModelSpec` entries (filename, display metadata, max size ceiling). It has no dependencies beyond Dart core.
- `model_repository.dart` owns all filesystem I/O. It lazily resolves the app-support `models/` directory via `path_provider`, performs streaming copy from user-selected source files into a `.part` temp file followed by atomic rename, validates each file against the GGUF magic bytes (`G G U F`) before committing, enforces per-model size ceilings, and falls back to pre-bundled Flutter assets on iOS/macOS for development. Public surface: `statusFor`/`statusAll`/`isComplete`/`resolvePathsIfComplete` returning `ModelStatus` objects plus `importModel` emitting `ModelImportProgress` events; errors are thrown as `ModelImportException`. A top-level `formatBytes` helper is re-exported for UI reuse.
- `ui/model_config_screen.dart` is a stateful Flutter page that depends only on the two model packages above plus `file_selector` and `path_provider`. It injects `ModelRepository` via constructor for testability, drives import/delete flows, and renders per-model cards with progress bars and status dots. Dependency direction is strictly one-way: UI → repository → catalog; the repository never imports Flutter code.
