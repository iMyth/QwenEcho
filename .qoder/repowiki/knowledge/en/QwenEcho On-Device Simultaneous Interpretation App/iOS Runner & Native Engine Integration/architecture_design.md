Standard Flutter iOS scaffold with three layers:
- `Runner/` — UIKit app entry. `AppDelegate.swift` subclasses `FlutterAppDelegate` and implements `FlutterImplicitEngineDelegate.didInitializeImplicitFlutterEngine`, where `GeneratedPluginRegistrant.register` wires all Dart plugins into the implicit engine. `SceneDelegate.swift` subclasses `FlutterSceneDelegate` without overrides so Flutter owns the full scene lifecycle; `Info.plist` declares a single `UIWindowScene` whose delegate is `$(PRODUCT_MODULE_NAME).SceneDelegate`.
- `Flutter/` — Flutter tooling artifacts (`Flutter.podspec`, `Debug.xcconfig`, `Release.xcconfig`, `Generated.xcconfig`) generated by `flutter create`; linking of the Flutter engine itself is handled by CocoaPods via these configs rather than hand-edited Xcode settings.
- `Pods/` + `Podfile` — dependency resolution. The Podfile pins `platform :ios, '16.0'`, disables CocoaPods analytics, calls `flutter_ios_podfile_setup` / `flutter_install_all_ios_pods`, and adds the local pod `qwen_echo_engine` from `../native` with `modular_headers => false`. A comment explicitly forbids `use_frameworks!` because the native engine must be linked as a static `.a` so its symbols are visible to the Runner binary for Dart FFI.

Dependency direction: Runner → Flutter engine (via CocoaPods) → qwen_echo_engine (local path pod, compiled C/C++ sources) → llama.cpp (git submodule under `../native`). The Podfile.lock records both the vendored `qwen_echo_engine (0.1.0)` and the resolved git commit `93d19d4b...` of the native engine, which in turn pulls in llama.cpp.
