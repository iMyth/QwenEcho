Each HAL capability is exposed as one C header in `native/hal/` (`hal_accelerator.h`, `hal_audio.h`, `hal_memory.h`, `hal_thermal.h`, `hal_thread.h`) declaring opaque handle types plus `hal_*` functions with `extern "C"` linkage so Swift/Kotlin can call them directly. Platform-specific implementations live under `android/` and `ios/` subdirectories using the same filename prefix (e.g. `hal_audio_android.c`, `hal_audio_ios.m`). The dependency direction is strictly one-way: platform sources include only the shared headers; there are no cross-OS includes or shared `.c` files between the two trees. Callback-based APIs (`hal_audio_capture_callback_t`, `hal_thermal_callback_t`) deliver real-time data from OS threads back into the engine, while stateful resources (`AcceleratorContext`, `AudioCapture`) follow a create/start/stop/destroy lifecycle. Accelerator backend selection (NNAPI/Vulkan on Android, CoreML/Metal on iOS) and CPU fallback are encapsulated inside `hal_accelerator_create` rather than exposed to callers.
